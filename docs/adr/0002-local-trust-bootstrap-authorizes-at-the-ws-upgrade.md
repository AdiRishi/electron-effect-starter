# Local Trust Bootstrap, Authorized At The WS Upgrade

The shell mints one random bootstrap token per process and hands it to the spawned server **over inherited file descriptor 3**, not argv or env. This is deliberate, not exotic plumbing: argv and environment blocks are readable by other local processes on many platforms, while an inherited fd exists only in the child. The spawn config also _clears_ the `APP_BOOTSTRAP_TOKEN`/`APP_SERVER_PORT` env fallbacks so a developer's shell cannot silently override the envelope. Clients (the renderer via the bridge, or a plain browser via `POST /api/auth/bootstrap/bearer`) exchange that token for an opaque bearer session, and the bearer is checked **once, at the `/ws` upgrade**.

There is deliberately no per-RPC authorization. T3 Code checks scopes inside individual RPC handlers; this starter has exactly one trust level — "paired with the local server" — so per-handler checks would be ceremony. The RPC contracts still declare `EnvironmentAuthorizationError` in their error channels: that is not dead weight but forward-compatibility, so clients are already typed for the day per-RPC authorization is added. Likewise the bearer store is in-memory with no expiry and plain string comparison — proportionate to a loopback-only server with process-lifetime sessions, and the same posture T3 Code takes for its local bootstrap token.
